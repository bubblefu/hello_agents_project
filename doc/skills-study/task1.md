# Agent Skills 学习笔记

## 一、核心概念：Agent Skills 是什么

### 内容整理

Agent Skills 是一种标准化的程序性知识封装格式，是跨 AI 应用通用的开放标准。它的核心是将特定场景下的领域知识、操作 SOP（标准作业程序）打包封装，让智能体能够掌握 “如何完成具体任务” 的方法，无需用户每次重复输入相同的要求。

它的核心价值是让智能体更自主：将重复的提示、规范、工作流打包成可复用的技能包，让智能体自动识别并应用，避免每次对话都要重新描述任务要求。

### 个人思考

这正好解决了我之前的核心痛点：之前每次和 Agent 做同类型的任务，都要把任务要求、规范、步骤重新说一遍，非常繁琐。而 Skill 就是把这些重复的内容打包成一个可复用的包，让 Agent 自动知道这类任务该怎么做，真正实现我想要的 “更自主的 Agent”，不用我每次都重复强调要求，也不用一直和 LLM 反复对话来明确目标。

## 二、核心机制：渐进式披露（Progressive Disclosure）

### 内容整理

这是 Agent Skills 最核心的创新，解决了传统模式下的上下文爆炸问题 —— 无需把所有技能的全部细节一次性加载到上下文窗口，而是智能体按需逐步加载，既不遗漏必要信息，又避免上下文过载，大幅降低 token 消耗和成本。

整个机制分为三层：

1. **第一层：元数据（Metadata）**
每个 Skill 都有独立的文件夹，核心是`SKILL.md`文件，文件开头必须是 YAML 格式的 Frontmatter，定义技能的基本信息（比如 name、description）。
智能体启动时，只会扫描所有已安装 Skill 的 Frontmatter 部分，把这些元数据加载到上下文。每个 Skill 的元数据仅消耗约 100 个 token，即使安装 50 个 Skill，初始上下文消耗也只有约 5000token，非常轻量化。

2. **第二层：技能主体（Instructions）**
当智能体分析用户请求，判断某个 Skill 和当前任务相关时，才会读取该 Skill 完整的`SKILL.md`文件，把详细的操作指令、注意事项、示例等加载到上下文。这部分内容的 token 消耗取决于指令复杂度，通常在 1000-5000token 之间，足够支撑完成任务的全部上下文。

3. **第三层：附加资源（Scripts & References）**
对于复杂 Skill，`SKILL.md`可以引用同文件夹下的其他文件：可执行脚本、参考文档、模板文件、图标资源等。智能体只有在真正需要用到这些资源的时候，才会加载 / 执行它们。
比如一个 PDF 处理 Skill，可以附带 PDF 解析脚本、表单填写指南、报告模板，只有当需要解析 PDF 的时候才执行对应脚本，需要填表的时候才加载表单指南，不会一开始就把这些内容都塞到上下文里。

这种设计带来两个关键优势：

- **无限知识容量**：可以附带超大的参考文件、数据集，智能体通过执行脚本来访问数据，无需将整个文件加载到上下文，突破了上下文窗口的限制。

- **确定性执行**：复杂的计算、数据转换、格式解析等任务交给代码执行，避免了 LLM 生成过程中的不确定性和幻觉问题。

### 个人思考

这正好解答了我之前的核心疑问！我之前一直疑惑：

> MD 中的每一个 name 和 description 都会作为 “token” 从一开始就被输入到 LLM 中，让 LLM 知道有这些 tool 可能被使用到。直到 LLM 判定需要使用哪个 tool 的时候，才去 [SKILL.md](SKILL.md) 中找该 tool 对应的其他信息，我说的对么？ 是让 LLM 还是 agent 读 md 文档？
> 
> 

现在我完全理清了：我的基础理解是完全正确的！

- 一开始加载到 LLM 上下文的，就是所有 Skill 的 Frontmatter 里的 name 和 description，也就是每个 Skill 的基础信息，让 LLM 知道有这些 Skill 可用。

- 而读取 MD 文档、管理加载节奏的主体是**Agent 框架**，不是 LLM 自己去读文件。Agent 负责扫描本地的 Skill 文件夹，调度资源加载：一开始只给 LLM 轻量化的元数据，当 LLM 判断当前任务需要用到某个 Skill 的时候，Agent 再去读取这个 Skill 的完整 MD 文件，甚至后续的附加资源，再把这些内容喂给 LLM，让 LLM 获得详细的操作指导。

这就是 Skill 最巧妙的地方，既让 LLM 知道有哪些能力可用，又不会一开始就把所有细节都塞进去，完美解决了上下文过载的问题。

## 三、Skill 的文件结构规范

### 内容整理

Skill 的文件结构有统一的规范，方便不同的 AI 平台识别和加载：

1. **文件夹命名**：Skill 的文件夹名使用小写字母，单词之间用连字符`-`连接，和 Skill 的 name 保持一致，比如`analyzing-marketing-campaign`。

2. **核心文件**：每个 Skill 文件夹必须包含`SKILL.md`文件，这是 Skill 的核心入口，文件开头必须有 YAML Frontmatter 来定义元数据。

3. **附加资源**：同文件夹下可以存放其他辅助文件：

    - 通常会把参考文档放在`references`文件夹里；

    - 可执行脚本：比如`.py`脚本，每个脚本实现一个具体的功能（比如表单检查、PDF 转图片等），`SKILL.md`里会定义这些脚本的作用和执行时机；

    - 其他资源：图标、图片、模板文件、数据集等，都可以放在文件夹内，供 Skill 按需调用。

4. **打包与安装**：整个 Skill 文件夹可以打包成压缩包，上传到支持 Skill 的 AI IDE / 平台中，比如 Claude AI 的 My Skill 功能。上传后平台会自动识别这个 Skill，后续用户提问时，LLM 会匹配问题中的关键词，自动触发对应的 Skill。

### 个人思考

这个基于文件系统的规范非常直观易用，完全符合我之前的理解：Skill 可以引用同文件夹下的所有文件，我可以把一个复杂任务需要的所有东西都打包到一个文件夹里，比如做营销活动分析的 Skill，我可以把 Excel 处理脚本、PPT 生成模板、行业参考文档都放进去，打包成一个包，上传之后就可以反复复用，非常方便。

而且 Skill 的能力不止是提示词，它还能管理脚本，决定这些脚本什么时候执行，这就让 Skill 的能力边界非常宽，不仅能指导 LLM 怎么做，还能直接用代码来完成那些 LLM 做不好的复杂计算、数据处理工作。

## 四、Skill 的适用场景

### 内容整理

当你遇到以下场景时，就适合创建一个 Skill：

1. 你需要重复做同类型的任务，每次都要输入相同的提示、规范、步骤，封装成 Skill 可以避免重复劳动。

2. 你想让你的 Agent 更自主，让 Agent 自动知道这类任务该怎么做，不用你每次都把要求重新说一遍。

3. 你需要封装复杂的工作流，比如结合多个工具、多个子任务的场景，Skill 可以把整个工作流的 SOP 打包进去，指导 Agent 一步步完成任务。

吴恩达课程中提到的实践方向：

1. 开发两个基础 Skill：内容创作 Skill、数据分析工作流 Skill，覆盖两类最常见的办公场景。

2. 开发营销活动分析 Skill，结合预构建的 Excel、PowerPoint 相关 Skill，实现营销数据的处理和报告生成的全流程自动化。

3. 可以用 Claude API 来测试开发好的 Skill，用 Claude Code 来对 Skill 里的脚本做代码 Review 和测试，保证 Skill 的可靠性。

4. 还可以结合 Claude Agent SDK 搭建研究用的 Agent，用 Skill 来整合研究的结果，形成完整的研究工作流。

### 个人思考

我之前设想的，Agent 可以结合 Skill 和 subagent 搭建复杂的自主工作流，现在看来完全可以实现：
Skill 提供了整个任务的 SOP，告诉 Agent 这个任务要分几步，每一步要做什么，然后 Agent 就可以把不同的子任务委托给 subagent 去完成，同时 Skill 里的脚本可以处理那些复杂的数据处理工作，不用 LLM 自己来做，既高效又能避免幻觉。

比如我要做一个市场研究的任务，我可以做一个研究 Skill，里面定义了：第一步要收集行业数据，第二步要做竞品分析，第三步要生成标准化的研究报告。然后 Agent 就可以把收集数据的任务交给负责搜索的 subagent，把分析的任务交给数据分析的 subagent，最后用 Skill 里的报告模板生成最终的文档。整个过程我只需要说一句 “帮我做一下 XX 行业的市场研究”，剩下的 Agent 自己就会按照 Skill 的指导完成，不用我一步步指挥，这就是我想要的完全自主的 Agent 工作流！

## 五、Skill 的适配环境

### 内容整理

目前海外已经有很多环境支持 Agent Skills，包括 Claude AI、Codex、Gemini CLI 等编程 / Agent 平台。

国内也已经有不少平台适配了这个开放标准，包括：

1. **腾讯系**：

    - 腾讯 CodeBuddy AI IDE：全面支持 Skills 功能，支持全局 Skill 和项目级 Skill；

    - 腾讯 Skill Hub：国内的 AI Skills 社区，针对国内用户优化，收录了超过 1.3 万个 Skill。

2. **字节系**：

    - 字节 TRAE AI IDE：已经上线 Skills 功能，支持自定义 Skill 包；

    - 火山引擎 Find Skill：Agent 技能广场，支持基于`SKILL.md`的 Skill 开发、二次开发和分发。

3. **阿里系**：

    - 阿里云通义 CoPAW：个人智能体工作台，支持用户添加自定义 Skills，通过`Skill.md`扩展能力；

    - 通义 qwencode：终端 AI 工具，已经支持 Agent Skills，使用方式和 Claude Skill 完全兼容。

4. **第三方平台**：

    - skill0：国内特赞团队推出的 Skills 市场，基于开放标准`SKILL.md`，目前已经收录了 423 个 Skill。
