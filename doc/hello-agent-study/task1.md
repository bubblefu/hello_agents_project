## 1. 笔记
- 关于智能体到底是什么？ 智能体和大模型有什么区别？
  - 我的理解： 
    - 智能体是一个AI产品或者系统，它和大模型是两个物种。智能体可以使用大模型，它利用了大模型的推理能力，结合一些tools，它也有记忆能力和规划能力，更像是复仇者联盟中 钢铁侠的虚拟管家贾维斯的形象。
    - 而大模型只是能思考，能推理问题，但无法做出比较真实的行动。大模型更加的工具化一些。
  
  - 举例：和LLM对话：我想喝咖啡，给我买个咖啡吧。  LLM会回答，如果你想喝到咖啡，你要怎么做12345等等。但绝不可能给你去买咖啡；  如果和一个智能体对话：给我买个咖啡吧。   智能体会判断：主人要买咖啡，需要线上购买，需要我能为他下单，需要我下的咖啡单最好是他喜欢喝常喝的品类；我去工具库里去找能做这个事的工具，比如我可以调美团外卖的API，我就能调用它，去完成线上下单甚至付款的事情。然后主人真的收到了咖啡外卖。
  - **总之**：智能体 = 大模型（思考） + 工具（行动） + 记忆（上下文） + 规划（策略）
  - 我希望一个agent可以自主地帮我解决一些问题


- 关于agent的运行机制
  - 循环、循环 还是循环
    - agent真正执行一次工作，在代码里都是一个for循环。即代码都在这个for循环里，次数可能是若干次或者判定提前结束。
    - 为什么是循环呢？————因为在跟环境不断地进行交互。
  - 如何交互？
    - 通过感知——思考——行动——观察——感知.....的多次循环逐步地推进任务
    - 简单的理解就有个类比：比如一个嵌入式恒温系统，这不涉及大模型的推理，要解决的任务也很单一，但逻辑是一样的————该系统的温度传感器一直监控环境温度，外部环境变化的时候，传感器传到芯片中的温度变化了，芯片里的判定系统说环境低了（比如），它发送升温指令给它外部的温度系统（比如燃气），多烧一些，让该恒温系统给环境提供更多热能，维持环境还处在恒温的状态下。 同时监控环境温度有没有恢复如初，还是反而变高了。再根据这些进行调节。
  - LLM agent的运行机制，名词概念：
    - 感知 (Perception)	接收环境输入（用户指令/工具返回结果）	
    - 思考 (Thought)	LLM内部推理，包含规划和工具选择	
    - 行动 (Action)	执行具体操作（调用工具）	
    - 观察 (Observation)

## 3. 第一章习题：
1. A B不是智能体。C有智能体的意思，但还不足够。 D是智能体
   1. A 是一个非常厉害的计算机器，它本质上是一个工具，它听指令办事，但不自主
   2. B 我觉得比较接近智能体了，但还不是自主的AI系统。它本质上是一个嵌入式系统，在极短地时间内进行响应调节，但它做出响应的依据是程序设定好的if else。
   3. C AlphaGo本质应该是依托于强化学习，进行多次地自己学习、试错，并不断反馈，然后调整思路。我觉得它体现的思想可以应用在智能体产品中，但是AlphaGo本身不能算作智能体。
   4. D 确实是智能体。它不仅有一个具体目标：解决客户的不同投诉问题。还能在过程中，根据用户的输入进行不同的反馈。
2. 答：
- 所谓PEAS模型
    - 性能度量(Performance)：评估智能体成功与否的标准
    1. 用户的目标达成率：是否在预期时间内达到减脂、增肌等目标
    2. 安全性指标：运动损伤发生率，过度训练预警准确率
    3. 用户依从性：训练计划是否按计划完成，完成多少
    4. 交互质量、语音指导的及时性、纠正动作准确性
    
    - 环境(Environment)：智能体运作的场景与条件
    1. 物理环境：健身房、户外、家里
    2. 环境数据：心率、血氧、运动姿态；历史训练记录数据库、运动科学知识库
    3. 交互环境：语音交互系统 、手机APP界面、手表等穿戴设备反馈系统
    - 执行器(Actuators)：智能体影响环境的方式
    1. 语音输出系统：实时指导、动作纠正提醒
    2. 视觉反馈：APP训练计划调整、数据可视化
    3. 振动系统
    4. 计划调整机制
    5. 预警系统：异常状态提醒
    - 传感器(Sensors)：智能体获取环境信息的渠道
    1. 生理检测指标：心率、血氧传感器、加速度计、陀螺仪等监控设备
    2. 用户输入：语音指令识别、触摸屏交互
    3. 用户历史数据、实时数据流接收

- 这个环境有哪些特性呢 
    1. 部分可观察：需要通过多种传感器间接推断用户状态
    2. 动态性：需要实时响应生理状态变化
    3. 随机性：需要处理用户个体差异和外部不确定性
    4. 延续性：需要长期跟踪和积累用户数据
    5. 连续性：需要在连续时空范围内做出决策


3. workflow vs agent
- workflow的优点是流程明确，责任清晰，有问题可以快速找到问题点和责任人；搭建链路也明确，代码量和后期维护可能不会很多；缺点是缺少鲁棒性，比如后边调整退款规则的话，很多地方的代码就得重构；还有一点是规则会显得非常死板，出现新的客户情况的话也许没有好办法。
- agent 的优点是能做出个性化的决策；并且可以不断地学习改进决策方案，不断提升有效性；不需要很多人工干预，能处理更复杂的案例；用户体验好；agent的缺点是 不可解释，难以解释具体决策原因，审查困难；系统初期很可能不好用，会有误判，用户不会让你去学习的；开发成本高；依赖数据质量
- workflow更适应：初期使用这个链路，快速搭建起来；如果法规严格的话也适合；没数据的话也适合
- agent更适合：商品种类特别多；数据非常丰富；愿意投入资源构建可进化的智能系统
- 作为电商公司负责人：如果是小型公司，优先才用workflow，快速上线，保证基本服务，积累数据，后边再插入agent；如果是大型公司，逐步引入agent，关键节点保留人工审核或者规则兜底

- 混合系统：
 workflow处理常规情况；agent处理负责情况；人工复核兜底 

4. 暂时这样，再想想
- 关于记忆功能，核心是将用户偏好从对话中提取并持久化存储。可以在工具层增加一个 save_preference 工具，当用户表达喜好时调用它存入数据库。同时在每次循环构建 Prompt 时，先从数据库读取该用户的历史偏好，拼接到 System Prompt 中，这样模型在 Thought 阶段就会自然考虑到用户喜欢历史文化或预算限制，生成的 Action 也会更精准，实现跨会话的记忆延续。
- 针对门票售罄的备选方案，关键在于工具返回的信息结构要包含余票状态。当 Observation 显示某景点售罄时，Agent 在下一轮 Thought 中应识别到这一负面反馈，自动调整参数再次调用同一工具，比如传入 exclude_sold_out=true 或要求推荐同类替代景点。这不需要改变主循环结构，而是依靠模型对 Observation 的理解来动态调整 Action 的输入参数，实现自动 fallback。
- 对于连续拒绝三次后的反思，需要在 Agent 运行状态中维护一个拒绝计数器，而不仅仅依赖 Prompt 历史。当计数器达到 3 时，动态修改 System Prompt，插入一条强指令要求模型暂停推荐，转而执行反思动作。可以新增一个 reflect 工具，让模型调用它来分析之前的失败原因，或者强制模型在 Thought 中输出策略调整计划，比如从推荐具体景点转变为询问用户具体不满的原因，从而打破无效推荐的循环。
5. 以医疗诊断助手为例。系统 1 负责快速处理非结构化数据，比如通过自然语言处理快速提取患者描述的症状关键词，或者通过图像识别快速判断 X 光片是否有明显异常，这就像医生的直觉反应，速度很快但可能不够精确。       系统 2 负责慢速的逻辑推理，比如根据系统 1 提供的症状线索，结合医学知识图谱进行鉴别诊断，检查药物之间是否有冲突，验证诊断结果是否符合临床指南，并生成可解释的诊断报告。两者协同工作时，系统 1 先给出初步的可能疾病列表和建议检查项目，系统 2 再接手对这些建议进行逻辑验证和排除，如果发现矛盾会要求系统 1 重新关注某些细节，最终由系统 2 输出确信的诊疗方案，这样既保证了响应速度又确保了医疗安全。

6. 大模型智能体产生幻觉是因为它本质是概率生成模型，基于训练数据预测下一个字，而不是真正理解事实，数据里的错误或者为了强行回答都可能导致编造信息。如果不设置最大循环次数，当工具调用失败时，智能体可能会无限重试，陷入死循环，导致 API 费用和时间的巨大浪费，甚至程序卡死。评估智能体不能只用准确率，还要看效率比如消耗了多少 token 和时间，鲁棒性比如出错后能否恢复，安全性以及推理过程是否合理，综合这些才能判断真正的智能程度。
